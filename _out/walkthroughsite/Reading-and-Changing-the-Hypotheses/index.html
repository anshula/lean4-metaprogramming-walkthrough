Reading and Changing the Hypotheses

All of the exercises in this section and the next are going to build up to writing an 'assumption' tactic ‚Äî one that looks at all the hypotheses, and if any matches the goal, successfully proves it using that hypothesis.

Reading All Hypotheses

So now we can read and modify the goal of a theorem.

What about the hypotheses?

All the hypotheses are stored in the local context, which is retrieved through getLCtx.

Some of them are not ones a human would think about when solving a theorem, that is, they are ‚Äúimplementation details‚Äù (e.g. the name of the theorem itself) and we skip them.

elab "print_hypotheses" : tactic => do let mut output : MessageData := .ofFormat .nil for ldecl in ‚Üê getLCtx do if ldecl.isImplementationDetail then continue let hypName := ldecl.userName let hypType := ldecl.type let message := m!"Name: '{hypName}' Type: '{hypType}'\n" output := .compose output message logInfo output

The new hypotheses are actually associated to the current goal.

(Quickly) Reading Newly-Added Hypotheses

We can also fix the same issue by just adding withMainContext to the beginning of the tactic.

elab "print_hypotheses''" : tactic => withMainContext do for ldecl in ‚Üê getLCtx do if ldecl.isImplementationDetail then continue let hyp_name := ldecl.userName let hyp_type := ldecl.type logInfo m!"Name: '{hyp_name}' Type: '{hyp_type}' "

Why does this work?

If you run a tactic, Lean works with the initial context of the theorem.

If you run withMainContext, then Lean gets the first goal (that is, the main metavariable), and adds in all of its context, and works with that.

We did that manually earlier by calling goal ‚Üê getMainGoal and then (‚Üê goal.getDecl).lctx. But withMainContext adds in all the right stuff for us.

Extracting Code into a Separate Definition

At this point, the tactics have gotten longer, and before we start adding onto this tactic in the next section, we might want to move out parts of the tactic into bits we can reuse in other tactics.

We can do this as long as we move out the side-effect-causing tasks into a function wrapped into a monad, either MetaM or TacticM. (We‚Äôll get into the differences between them later).

def printHypotheses : TacticM Unit := do let goal ‚Üê getMainGoal for ldecl in ‚Üê getLCtx do if ldecl.isImplementationDetail then continue let hypName := ldecl.userName let hypType := ldecl.type logInfo m!"Name: '{hypName}' Type: '{hypType}'"

We can still call print_hypotheses as normal as long as we also create an elab that tells Lean which function to call when the user types print_hypotheses.

elab "print_hypotheses" : tactic => do printHypotheses

Returning Values

The function we used above looked like

def printHypotheses : TacticM Unit := do
	...

In general, if we ever write a tactic that doesn‚Äôt return anything (and perhaps, just prints something to the screen), it will have the Unit return type.

The Unit serves the purpose that void does in other programming languages ‚Äî it tells us the function isn‚Äôt going to return anything interesting.

Getting a Particular Hypothesis

What if we want to print a particular hypothesis?

Then we need return values. So far we've just printing to the screen, and been returning Unit. What if we actually want to return values, so that you can use them in another method? We can do that by wrapping the return value (in this case, a LocalDecl) in a monad (in this case, TacticM).

def getHypothesisByName (h : Name) : TacticM LocalDecl := withMainContext do let goal ‚Üê getMainGoal -- the dynamically generated hypotheses are associated with this particular goal for ldecl in (‚Üê goal.getDecl).lctx do if ldecl.isImplementationDetail then continue if ldecl.userName == h then return ldecl throwError m!"No hypothesis by the name {h}."

The above works for if you want to call this method within a metaprogram.

But if you want a similar function to call within a theorem, you can only pass in "terms" not names directly. Here we do some pre-processing on the term to get a name.

def getHypothesisByFVarId (h : FVarId) : TacticM LocalDecl := withMainContext do let goal ‚Üê getMainGoal -- the dynamically generated hypotheses are associated with this particular goal for ldecl in (‚Üê goal.getDecl).lctx do if ldecl.isImplementationDetail then continue if ldecl.fvarId == h then return ldecl throwError m!"No hypothesis with fvarId {h.name}." def getHypothesisByTerm (h : TSyntax `term) : TacticM LocalDecl := withMainContext do let fvarId ‚Üê getFVarId h getHypothesisByFVarId fvarId elab "print_hypothesis_by_name" h:term : tactic => withMainContext do let decl ‚Üê getHypothesisByTerm h logInfo decl.type

Writing Tactics with MetaM vs TacticM

Our printHypotheses function worked when we had it return a TacticM.

But if we change it to use MetaM, it fails.

def printHypotheses' : MetaM Unit := do let goal ‚Üê type mismatch getMainGoal has type TacticM MVarId : Type but is expected to have type MetaM ?m.11858 : TypegetMainGoal failed to construct 'ForIn' instance for collection ?m.13456 and monad ReaderT Meta.Context (StateRefT' IO.RealWorld Meta.State CoreM)for ldecl in invalid field notation, type is not of the form (C ...) where C is a constant __do_lift‚úù has type ?m.13434 goal(‚Üê invalid field notation, type is not of the form (C ...) where C is a constant goal has type ?m.11847goal.getDecl).lctx do if ldecl.isImplementationDetail then continue let hypName := ldecl.userName let hypType := ldecl.type logInfo m!"Name: '{hypName}' Type: '{hypType}'"

That‚Äôs because working within TacticM gives us access to the current goals ‚Äî if we try to access the goals outside TacticM, we get an error.

In fact, TacticM has access to everything MetaM has access too, plus goals.

Why ever use MetaM?

So if TacticM is just MetaM but more, why would we ever use MetaM?

The following code works if you use MetaM‚Ä¶

/-- Check the context for a theorem named `riemannHypothesis`. -/ def lookIntoEnvironment : MetaM Unit := do let env ‚Üê getEnv dbg_trace (env.contains `riemannHypothesis) false #eval lookIntoEnvironment

‚Ä¶ but fails if you use TacticM.

/-- Check the context for a theorem named `riemannHypothesis`. -/ def lookIntoEnvironment' : TacticM Unit := do let env ‚Üê getEnv dbg_trace (env.contains `riemannHypothesis) unable to synthesize 'MonadEval' instance to adapt TacticM Unit to 'IO' or 'Command.CommandElabM'.#eval lookIntoEnvironment'

What‚Äôs going on?

If we ever want to check how our functions work using #eval statements, they need to implement the MetaEval class ‚Äî which allows output to be easily evaluated. MetaM implements this class, but TacticM doesn‚Äôt.

So, we ideally want to return something of type TacticM only when we really need it.

In summary, here are the perks of each:

  • MetaM lets you access most of the local context (including declared metavariables, therefore the name MetaM).

  • TacticM lets you access anything in MetaM, and also the current goals you have in the theorem the tactic is being used in (therefore the name TacticM)..

And here are the downsides:

  • We can‚Äôt use a MetaM function to peer into a goal ‚Äî MetaM doesn‚Äôt allow us access to a list of goals.

  • We can‚Äôt call a TacticM function from #eval.
