Adding Goals and Hypotheses

By the end of this section, you'll create a version of the apply tactic. This tactic "applies" a lemma which has a conclusion that matches the current goal, and updates the current goal accordingly.

Adding a Goal

We know how to manipulate a list of goals in Lean (for example, rotating the order of them). We will now look at modifying the tactic state beyond just manipulating the list of goals.

Creating a goal in Lean is really creating a metavariable (a variable whose value a.k.a proof isn‚Äôt known yet). Proving a goal in Lean is assigning a value a.k.a proof to that metavariable.

We can extract out the basics of goal creation into a helper tactic: createGoal.

open Lean Meta Elab Tactic def createGoal (goalType : Expr) : TacticM Unit := withMainContext do let goal ‚Üê mkFreshExprMVar goalType appendGoals [goal.mvarId!]

We can see this in action here. If we want our goal to be ‚Äúfind an instance of type Nat‚Äù, we create a metavariable with type Nat, like so:

elab "create_nat_goal" : tactic => do -- make the goal to find an instance of type "Nat" let goalType := (Expr.const ``Nat []) createGoal goalType example : 1 + 2 = 3 := ‚ä¢ 1 + 2 = 3 ‚ä¢ 1 + 2 = 3‚ä¢ ‚Ñï ‚ä¢ ‚Ñï All goals completed! üêô

If instead, we want to create a goal to be ‚Äúprove 0 = 0‚Äù, then we create a metavariable with type 0 = 0.

elab "create_reflexivity_goal" : tactic => do -- make the metavariable goal to prove that "0 = 0" let goalType ‚Üê mkEq (toExpr 0) (toExpr 0) createGoal goalType example : 1 + 2 = 3 := ‚ä¢ 1 + 2 = 3 ‚ä¢ 1 + 2 = 3‚ä¢ 0 = 0 ‚ä¢ 0 = 0; All goals completed! üêô

Until now, we had to be in an elab to decide what goal would be created.

But now, we can create a tactic that allows us to decide what the new goal is from within the proof itself.

elab "create_goal" t:term : tactic => do let e ‚Üê Term.elabTerm t none createGoal e example : 1 + 2 = 3 := ‚ä¢ 1 + 2 = 3 ‚ä¢ 1 + 2 = 3‚ä¢ Nat.Prime 2 ‚ä¢ Nat.Prime 2; All goals completed! üêô

Proving a Goal

Remember:

  • Creating a goal in Lean is really creating a metavariable (a variable whose value a.k.a proof isn‚Äôt known yet).

  • Proving a goal in Lean is assigning a value a.k.a proof to that metavariable.

So, we can prove a goal by passing in an expression for the term that matches the goal's type.

def proveGoal (proof : Expr) : TacticM Unit := withMainContext do (‚Üê getMainGoal).assign proof replaceMainGoal []

We can see this in action here:

Adding a Hypothesis

Similarly, we can extract out the basics of hypothesis creation into a helper tactic: createHypothesis.

def createHypothesis (hypType : Expr) (hypProof : Expr) (hypName := `h) : TacticM Unit := do let hyp : Hypothesis := { userName := hypName, type := hypType, value := hypProof } let (_, new_goal) ‚Üê (‚Üê getMainGoal).assertHypotheses ( List.toArray [hyp] ) setGoals [new_goal]

Here‚Äôs an example of adding a hypothesis that looks like h : ‚Ñï.

elab "create_nat_hypothesis" : tactic => do let hypType := Expr.const ``Nat [] -- type of Nat let hypProof := (toExpr 0) -- term of 0 createHypothesis hypType hypProof example : 1 + 2 = 3 := ‚ä¢ 1 + 2 = 3 h:‚Ñï‚ä¢ 1 + 2 = 3 All goals completed! üêô

Here‚Äôs an example of adding a hypothesis that is a proposition 0 = 0.

elab "create_reflexivity_hypothesis" : tactic => do
  let hypType ‚Üê mkEq (toExpr 0) (toExpr 0) -- statement that "0 = 0"
  let hypProof ‚Üê mkAppM ``Eq.refl #[toExpr 0] -- proof that "0 = 0"
  createHypothesis hypType hypProof

example : 1 + 2 = 3 := by

  simp

Did We Prove the Hypothesis?

Lean will technically let us create bogus hypotheses.

elab "create_bogus_hypothesis" : tactic => do let hypType ‚Üê mkEq (toExpr 0) (toExpr 1) -- statement that "0 = 1" let hypProof ‚Üê mkAppM ``Eq.refl #[toExpr 0] -- proof that "0 = 0" createHypothesis hypType hypProof declaration uses 'sorry'example : 0=1 := ‚ä¢ 0 = 1 h:0 = 1‚ä¢ 0 = 1 -- "0=1" is now a hypothesis All goals completed! üêô

But, if we actually try to use that hypothesis to prove a theorem, it will throw a low-level error in the kernel. A similar thing occurred previously with setGoals, where we could drop all active goals from the tactic state, but we got a kernel error if we tried to then finish the proof.

(kernel) declaration type mismatch, '_example' has type 0 = 0 but it is expected to have type 0 = 1example : 0=1 := ‚ä¢ 0 = 1 h:0 = 1‚ä¢ 0 = 1 All goals completed! üêô

Ideally, we want to catch this error earlier.

To do that, we want our tactic to check that the hypothesis statement matches the proof. That is, by propositions-as-types and proofs-as-terms, the inferred type of the hypProof should be hypType.

So let's modify our createHypothesis function to perform this check.

def createHypothesisGuarded (hypType : Expr) (hypProof : Expr) (hypName := `h): TacticM Unit := do unless ‚Üê isDefEq hypType (‚Üê inferType hypProof) do throwError "The proof used for creating the hypothesis does not match the expected type." let hyp : Hypothesis := { userName := hypName, type := hypType, value := hypProof } let (_, new_goal) ‚Üê (‚Üê getMainGoal).assertHypotheses ( List.toArray [hyp] ) setGoals [new_goal]

Here, the inferType function is used to deduce the type of hypProof (or the statement of the theorem it is a proof of, in the case of propositions). The isDefEq function is used to compare two expressions up to definitional equality; more will be said about this function in the later sections.

Adding the bogus hypothesis no longer works with the createHypothesisGuarded function.

elab "create_bogus_hypothesis'" : tactic => do let hypType ‚Üê mkEq (toExpr 0) (toExpr 1) -- statement that "0 = 1" let hypProof ‚Üê mkAppM ``Eq.refl #[toExpr 0] -- proof that "0 = 0" createHypothesisGuarded hypType hypProof example : 0=1 := ‚ä¢ 0 = 1 ‚ä¢ 0 = 1 assumption

Implementing the 'apply' Tactic

We typically make progress on a proof in Lean through tactics. These have the effect of replacing the current proof state with another, in a way that preserves provability (i.e., if the new set of goals is solvable, then so is the old one).

The converse is not true - it is possible to start from a solvable tactic state and go to an unsolvable one, for example by clearing an essential hypothesis or using backwards reasoning on the goal.

So far, we have written tactics that work on the list of active goals or add hypotheses to the local context. To better understand how tactics manipulate the proof state, let's write a simplified version of the apply tactic - the tactic that is routinely used in Lean for backwards reasoning, i.e., reasoning backwards from the target.

An Example

For example, suppose the goal is to prove that 2 ^ 3 is not a prime number.

And suppose we have a theorem called Nat.Prime.not_prime_pow, which says that if x and n are natural numbers and 2 ‚â§ n, then x ^ n is not a prime number.

A mathematician would say "to show that 2 ^ 3 is not prime, it is sufficient to show that 2 ‚â§ 3, by the result Nat.Prime.not_prime_pow." (except maybe for the last bit).

A Lean coder would say apply Nat.Prime.not_prime_pow.

Nat.Prime.not_prime_pow {x n : ‚Ñï} (hn : 2 ‚â§ n) : ¬¨Nat.Prime (x ^ n)#check Nat.Prime.not_prime_pow example : ¬¨ Nat.Prime (2 ^ 3) := ‚ä¢ ¬¨Nat.Prime (2 ^ 3) ‚ä¢ 2 ‚â§ 3 -- the goal is now to show that `2 ‚â§ 3` All goals completed! üêô

Implementation

We'll start with the scenario where we have a goal Q and a local hypothesis of type P ‚Üí Q.

h: P ‚Üí Q
======
‚ä¢ Q

After calling, apply, we want the goal to be P. (Because, indeed, it is sufficient to prove P.)

h: P ‚Üí Q
======
‚ä¢ P

Our high-level strategy is going to be to:

  • Check whether the hypothesis h is an implication, i.e., of the form P ‚Üí Q

h: P ‚Üí Q -- this is an implication
======
‚ä¢ Q

  • Check whether the conclusion of the hypothesis h matches the type of the current goal

h: P ‚Üí Q -- the conclusion of the hypothesis is Q
======
‚ä¢ Q -- the type of the current goal is Q

  • Create a new goal p of type P

h: P ‚Üí Q
======
‚ä¢ P -- we create a new goal
‚ä¢ Q

  • We can prove the goal "Q" assuming the hypothesis "P‚Üí Q" (called h) and assuming we've already proved the first goal "P" (called p). So, we can prove the goal "Q" by assigning it the value h p.

h: P ‚Üí Q
======
‚ä¢ P
-- ‚ä¢ Q -- we can prove this goal

  • We're done.

h: P ‚Üí Q
======
‚ä¢ P

To do this, we'll use a helper function createGoal' which unlike our previous createGoal, actually returns the new goal.

def createGoal' (goalType : Expr) : TacticM Expr := withMainContext do let goal ‚Üê mkFreshExprMVar goalType appendGoals [goal.mvarId!] return goal

Here is what this full strategy looks like translated into code. Parts of this code are deliberately sub-optimal; we'll see ways of improving it shortly.

And here is the tactic:

elab "apply_hypothesis" h:term : tactic => withMainContext do -- ensure that the hypothesis is an implication let hyp ‚Üê getHypothesisByTerm h guard hyp.type.isArrow -- extract implication's antecedent & consequent -- or throw error if the hypothesis type is not a .forall let .forallE _ P Q _ := hyp.type | unreachable! -- ensure that the conclusion of `h` matches the target unless Q == (‚Üê getMainTarget) do throwError m!"The type of the conclusion of {h} does not match the current target." -- create a new goal of type `P` let newGoal ‚Üê createGoal' P -- close off the current goal with `h newGoal` proveGoal (.app hyp.toExpr newGoal) example (h : Even 2 ‚Üí Even 4) : Even 4 := h:Even 2 ‚Üí Even 4‚ä¢ Even 4 h:Even 2 ‚Üí Even 4‚ä¢ Even 2 -- the goal is now `Even 2` All goals completed! üêô

However, our implementation has some shortcomings. Let's try to change the target to Even (2 * 2) instead of Even 4:

example (h : Even 2 ‚Üí Even 4) : Even (2 * 2) := h:Even 2 ‚Üí Even 4‚ä¢ Even (2 * 2) h:Even 2 ‚Üí Even 4‚ä¢ Even (2 * 2) -- a type mismatch error

We get a type mismatch error since Lean can't tell that Even 4 is the same as Even (2 * 2), even though they are equal by definition. This is because our comparison of expressions in the code uses "Boolean equality" (denoted ==), which is too strict for our purposes.

The error goes away when we switch to the coarser notion of "definitional equality" (checked by the function isDefEq):

elab "apply_hypothesis_defeq" h:term : tactic => withMainContext do -- ensure that the hypothesis is an implication let hyp ‚Üê getHypothesisByTerm h guard hyp.type.isArrow -- extract the implication's antecedent & consequent let .forallE _ P Q _ := hyp.type | unreachable! -- ensure that the conclusion of `h` matches the target unless ‚Üê isDefEq Q (‚Üê getMainTarget) do throwError m!"The type of the conclusion of {h} does not match the current target." -- create a new goal of type `P` let newGoal ‚Üê createGoal' P -- close off the current goal with `h newGoal` proveGoal (.app hyp.toExpr newGoal)

Now the example works as intended.

example (h : Even 2 ‚Üí Even 4) : Even (2 * 2) := h:Even 2 ‚Üí Even 4‚ä¢ Even (2 * 2) h:Even 2 ‚Üí Even 4‚ä¢ Even 2 -- the goal is now `Even 2` All goals completed! üêô

Implementing the 'apply' Tactic via Unification

As mentioned before, the function isDefEq does more than just checking for definitional equality - it also handles the unification (roughly, filling in holes) of expressions containing meta-variables (roughly, expressions with holes).

For example, consider the two expressions (7 * _) and (_ * 3), where the underscores indicate "holes" in the expressions. While these expressions are not equal by ==, they are by isDefEq, since they can be made equal by choosing the values for the holes appropriately (so that they both become 7 * 3). This is the idea behind unification, and the isDefEq function tries to fill in the holes as much as possible to make the two expressions match up.

elab "try_to_unify" a:term "and" b:term : tactic => do let a ‚Üê Term.elabTermAndSynthesize a none let b ‚Üê Term.elabTermAndSynthesize b none -- use isDefEq to compare expressions let unused variable `aEqB` note: this linter can be disabled with `set_option linter.unusedVariables false`aEqB ‚Üê isDefEq a b --... but the expressions still aren't exactly equal let aEqB := a == b logInfo m!"It is {aEqB} that expressions a and b are exactly equal (according to ==)." -- use instantiateMVars to find out how isDefEq filled the holes -- and fill them accordingly let a ‚Üê instantiateMVars a let b ‚Üê instantiateMVars a let aEqB := a == b logInfo m!"It is {aEqB} that expressions a and b are exactly equal, after using unification to fill holes." -- show vars after using unification to fill hole logInfo m!"This is a after filling holes: {a}" logInfo m!"This is b after filling holes: {b}" example : True := ‚ä¢ True ‚ä¢ True All goals completed! üêô

The word meta-variable was used earlier to refer to a goal in the tactic, and here is used to refer to a hole in an expression. However, the two senses of the word are essentially the same ‚Äî a goal can be thought of as a hole for a proof of the appropriate type.

We can implement the apply_hypothesis using a strategy involving unification:

  • We start with a hypothesis h of type H, and a goal of type Q.

h: H
======
‚ä¢ Q

  • Create a meta-variable ?P for the type of the new goal

h: H
======
‚ä¢ ?P
‚ä¢ Q

  • Check whether the type of the hypothesis unifies with the expression ?P ‚Üí Q (which really means anything ‚Üí Q).

h: H -- does this take the form (something ‚Üí Q)?
======
‚ä¢ ?P
‚ä¢ Q

  • If the unification succeeds and the meta-variable ?P is assigned the value P, create a meta-variable of type P for the new goal

h: P ‚Üí Q
======
‚ä¢ P -- it suffices to prove "P"
‚ä¢ Q

  • Assign the value h p to the goal of type "Q"

h: P ‚Üí Q
======
‚ä¢ P
-- ‚ä¢ Q -- we've proven this

  • We're done.

h: P ‚Üí Q
======
‚ä¢ P

With this approach, we no longer need to check if the hypothesis is an implication and explicitly extract its antecedent and consequent.

elab "apply_hypothesis_unif" h:term : tactic => withMainContext do -- get the hypothesis let hyp ‚Üê getHypothesisByTerm h -- call the current goal Q let Q ‚Üê getGoalType -- try to unify hypothesis with `P? ‚Üí Q` let P? ‚Üê mkFreshExprMVar none unless ‚Üê isDefEq hyp.type (‚Üê mkArrow P? Q) do throwError m!"The hypothesis is expected to be an implication with conclusion matching the current goal." -- fill in holes in `P` with what made it unify with `P?` let P ‚Üê instantiateMVars P? -- create a new goal of type `P` let newGoal ‚Üê createGoal' P -- close off the current goal with `h newGoal` proveGoal (.app hyp.toExpr newGoal)

And the tactic works the same as before.

example (h : Even 2 ‚Üí Even 4) : Even (2 * 2) := h:Even 2 ‚Üí Even 4‚ä¢ Even (2 * 2) h:Even 2 ‚Üí Even 4‚ä¢ Even 2 All goals completed! üêô

Generalizing the 'apply' Tactic

To finish off this chapter, we'll generalize our apply_hypothesis tactic to scenarios where the type of the argument is not just a single implication. (The Lean 4 Metaprogramming Book goes through this example in detail.)

For example, suppose we have a target like ¬¨Nat.Prime (2 * 3) that we want to prove by backwards reasoning using Nat.not_prime_mul (shown below), which says that the product of two numbers is not prime when those numbers are both not equal to 1.

Nat.not_prime_mul {a b : ‚Ñï} (a1 : a ‚â† 1) (b1 : b ‚â† 1) : ¬¨Nat.Prime (a * b)#check Nat.not_prime_mul

We would like our apply tactic to infer values of the implicit parameters a and b through unification and create two new goals corresponding to the remaining arguments: 2 ‚â† 1 and 3 ‚â† 1.

Lean contains utilities for taking expressions of the form P‚ÇÅ ‚Üí (P‚ÇÇ ‚Üí ... ‚Üí P‚Çñ ‚Üí (... ‚Üí Q)) and extracting the hypotheses #[P‚ÇÅ, P‚ÇÇ, ..., P‚Çñ, ...] along with the conclusion Q (which go under the fanciful name of "telescopes"). These turn out to be exactly what we need to implement a version of the apply tactic that does what we want.

Here is our rough strategy for implementing the tactic:

  • Infer the type of the hypothesis being applied

  • Using a telescope, obtain the list of meta-variables #[p‚ÇÅ, p‚ÇÇ, ..., p‚Çñ, ...] for the conditions along with the conclusion Q

  • Attempt to unify Q with the current target

  • Assign the value h p‚ÇÅ p‚ÇÇ ... p‚Çñ ... to the current goal, where h is the hypothesis being applied

  • Make the hypotheses #[P‚ÇÅ, P‚ÇÇ, ..., P‚Çñ, ...] the new targets

elab "apply_to_target" h:term : tactic => withMainContext do -- Get the hypothesis being applied let hypTerm ‚Üê elabTerm h none let hypType ‚Üê inferType hypTerm -- Obtain conditions and the conclusion using a telescope let (conditions, _, conclusion) ‚Üê forallMetaTelescope hypType -- Attempt to unify the conclusion with the main target unless ‚Üê isDefEq conclusion (‚Üê getGoalType) do throwError m!"The conclusion of the hypothesis {h} does not match with the current target." -- Fill holes in conditions via unification let conditions ‚Üê conditions.mapM instantiateMVars -- Set the conditions as the new goals appendGoals (conditions.toList.map Expr.mvarId!) -- Assign the conclusion to the current goal proveGoal (mkAppN hypTerm conditions) example : ¬¨ Nat.Prime (2 * 3) := ‚ä¢ ¬¨Nat.Prime (2 * 3) ‚ä¢ 2 ‚â† 1‚ä¢ 3 ‚â† 1 -- the two new goals are `2 ‚â† 1` and `3 ‚â† 1` all_goals All goals completed! üêô
