Reading and Changing the Goal

A First Tactic

Here is a super simple tactic: the do_nothing tactic.

open Lean Elab Tactic elab "do_nothing" : tactic => do return

We‚Äôll see that, indeed, this tactic changes nothing about the proof state. Hover over the end-of-line bubbles to see the goal state.

Reading the Current Goal

Now, let‚Äôs create a tactic print_goal that reads what the current goal is.

elab "print_goal" : tactic => do let goal ‚Üê getMainGoal logInfo goal

Let‚Äôs test the tactic:

example : 1+1=2 := ‚ä¢ 1 + 1 = 2 ‚ä¢ 1 + 1 = 2 -- 1+1=2 All goals completed! üêô

Reading All Goals

We've seen before that getMainGoal gives us the details of the current goal. The getUnsolvedGoals command gives us all the active goals in the tactic state.

Here is a tactic that lets us test this out.

elab "print_goals" : tactic => do let goals ‚Üê getUnsolvedGoals for goal in goals do logInfo goal example : 1 + 1 = 2 ‚àß 2 + 2 = 4 := ‚ä¢ 1 + 1 = 2 ‚àß 2 + 2 = 4 ‚ä¢ 1 + 1 = 2 ‚àß 2 + 2 = 4 ‚ä¢ 1 + 1 = 2‚ä¢ 2 + 2 = 4 ‚ä¢ 1 + 1 = 2‚ä¢ 2 + 2 = 4 all_goals All goals completed! üêô

And we get what we expect.

Switching Around Goals

Now we can read the goal. Let‚Äôs modify it.

The list of goals can also be modified with the setGoals command.

For example, here is an implementation of a rotate_goals tactic that reorders the goals to push the main goal to the end.

elab "rotate_goals" : tactic => do let goals ‚Üê getUnsolvedGoals setGoals goals.rotateLeft example : 1 + 1 = 2 ‚àß 2 + 2 = 4 := ‚ä¢ 1 + 1 = 2 ‚àß 2 + 2 = 4 ‚ä¢ 1 + 1 = 2‚ä¢ 2 + 2 = 4 ‚ä¢ 2 + 2 = 4‚ä¢ 1 + 1 = 2 -- the goals are now in a different order all_goals All goals completed! üêô

Deleting Goals

Now what happens if the user decides to use setGoals to just delete the list of active goals?

elab "clear_goals" : tactic => do setGoals [] (kernel) declaration has metavariables '_example'example : 1 + 1 = 2 ‚àß 2 + 2 = 4 := ‚ä¢ 1 + 1 = 2 ‚àß 2 + 2 = 4 ‚ä¢ 1 + 1 = 2‚ä¢ 2 + 2 = 4 All goals completed! üêô -- there are no goals here

Doing this indeed clears all the goals in the tactic state, but a low-level kernel error now pops up near the start of the declaration. So Lean can't be tricked into accepting an incomplete proof, and the responsibility of making sure no active goals get dropped is on the meta-programmer.

Modifying Goals

Let's write a tactic that modifies the goal in a more significant way.

Let‚Äôs write a tactic that turns a theorem into its contrapositive. First, let‚Äôs prove that a contrapositive tactic could work.

theorem ctrp {P Q : Prop} : (contra: ¬¨ Q ‚Üí ¬¨ P) ‚Üí (P ‚Üí Q) := P:PropQ:Prop‚ä¢ (¬¨Q ‚Üí ¬¨P) ‚Üí P ‚Üí Q P:PropQ:Proph:¬¨Q ‚Üí ¬¨P‚ä¢ P ‚Üí Q rwa [not_imp_notP:PropQ:Proph:P ‚Üí Q‚ä¢ P ‚Üí Q at h

Now, if we ever want to use this theorem, we can type apply ctrp.

example {P : Prop} : P ‚Üí True := P:Prop‚ä¢ P ‚Üí True P:Prop‚ä¢ ¬¨True ‚Üí ¬¨P All goals completed! üêô

But since the line apply ctrp is so cumbersome to write out, lets wrap it up into a one-word tactic called contrapos.

Writing a tactic ‚Äî what doesn‚Äôt work

Now, we‚Äôve been using elab (name) : tactic => ... to create tactics. But elab is not very convenient to use if we are just planning on conglomerating a bunch of already-existing Lean tactics.

That is, the following code doesn't work:

elab "contrapos" : tactic => do unknown identifier 'apply'apply ctrp -- throws error!

That‚Äôs because there are a bunch of low-level configuration options you need to send to apply if you‚Äôre going to call it from within a tactic, and that‚Äôs a bit of a pain.

Writing a tactic ‚Äî what does work

So, instead, when we want to conglomerate existing Lean tactics, we use macro:

macro "contrapos" : tactic => `(tactic| apply ctrp)

We can test it out.

example {P : Prop} : P ‚Üí True := P:Prop‚ä¢ P ‚Üí True P:Prop‚ä¢ ¬¨True ‚Üí ¬¨P All goals completed! üêô

So that‚Äôs ‚Äúelaboration‚Äù and ‚Äúmacros‚Äù ‚Äî we can use either to write Lean tactics.

Writing Tactics with macro vs elab

We noticed that apply works easily within a macro, but not within an elab. It‚Äôs the same with lots of Lean tactics, for example, sorry.

To write sorry in a theorem, you just have to write sorry.

theorem declaration uses 'sorry'my_sorry_theorem : True := sorry

To write sorry in a macro, you can easily access the "theorem-proving" mode with tactic|.

macro "my_sorry_macro" : tactic => `(tactic| sorry)

To write sorry in an elab, you have to get a bit lower level, and use ‚ÄúadmitGoal‚Äù and pass it an argument.

elab "my_sorry_elab" : tactic => do let goal ‚Üê getMainGoal admitGoal goal

In general, macro lets you work at a higher level than elab, but you get less control.

As such, if your tactic doesn‚Äôt have any real programming logic, and is just conglomerating some existing tactics, as above, you should use macro.

If there‚Äôs a task at hand that requires some level of customization, you should use elab.

Providing Arguments to Tactics

We can also provide arguments to a macro or elab. Here‚Äôs an example where arguments come in handy.

Say we have P ‚Üí Q ‚Üí True.

It‚Äôs quick to contrapose Q and True:

example {P Q : Prop} : P ‚Üí Q ‚Üí True := P:PropQ:Prop‚ä¢ P ‚Üí Q ‚Üí True P:PropQ:Propp:P‚ä¢ Q ‚Üí True P:PropQ:Propp:P‚ä¢ ¬¨True ‚Üí ¬¨Q -- `Q` and `True` have been contraposed All goals completed! üêô

But more annoying to contrapose P and True.

example {P Q : Prop} : P ‚Üí Q ‚Üí True := P:PropQ:Prop‚ä¢ P ‚Üí Q ‚Üí True P:PropQ:Propp:Pq:Q‚ä¢ True P:PropQ:Propq:Q‚ä¢ P ‚Üí True P:PropQ:Propq:Q‚ä¢ ¬¨True ‚Üí ¬¨P -- `P` and `True` have been contraposed All goals completed! üêô

Let‚Äôs create a tactic that will contrapose the conclusion with the given hypothesis h.

macro "contrapos_with" h:ident : tactic => `(tactic| (revert $h; contrapos) )

We can test it out.

example {P Q : Prop} : P ‚Üí Q ‚Üí True := P:PropQ:Prop‚ä¢ P ‚Üí Q ‚Üí True P:PropQ:Propp:Pq:Q‚ä¢ True P:PropQ:Propp:P‚ä¢ ¬¨True ‚Üí ¬¨Q -- `Q` and `True` have been contraposed All goals completed! üêô example {P Q : Prop} : P ‚Üí Q ‚Üí True := P:PropQ:Prop‚ä¢ P ‚Üí Q ‚Üí True P:PropQ:Propp:Pq:Q‚ä¢ True P:PropQ:Propq:Q‚ä¢ ¬¨True ‚Üí ¬¨P -- `P` and `True` have been contraposed All goals completed! üêô

And it works as expected.

Providing tactics as arguments to tactics

So above, we provided a proposition (technically, an identifier pointing to a proposition) as an argument.

We can also provide a tactic as an argument. For example, this example from the Lean 4 Metaprogramming Book takes two tactics, runs the first one (which potentially creates more goals), then runs the second one on all the goals.

So without the tactic, you might have to do something like this:

example: 1=1 ‚àß 2=2 := ‚ä¢ 1 = 1 ‚àß 2 = 2 ‚ä¢ 1 = 1‚ä¢ 2 = 2 -- split into two goals: 1 = 1 and 2 = 2 ‚ä¢ 2 = 2; All goals completed! üêô -- solve each one

But then you create this tactic‚Ä¶

macro "and_then" a:tactic b:tactic : tactic => `(tactic| ($a:tactic; all_goals $b:tactic) )

‚Ä¶And you can do this:

example: 1=1 ‚àß 2=2 := ‚ä¢ 1 = 1 ‚àß 2 = 2 and_then ‚ä¢ 1 = 1‚ä¢ 2 = 2 All goals completed! üêô

Creating Intuitive Syntax for Tactics

Instead of writing and_then constructor rfl, it might be more intuitive to write the above tactic as constructor and_then rfl.

This is where it‚Äôs helpful to create a syntax rule.

syntax tactic " and_then " tactic : tactic macro_rules | `(tactic| $a:tactic and_then $b:tactic) => `(tactic| and_then $a $b)

Now we can write this tactic much more intuitively:

example: 1 = 1 ‚àß 2 = 2 := ‚ä¢ 1 = 1 ‚àß 2 = 2 ‚ä¢ 1 = 1‚ä¢ 2 = 2 and_then All goals completed! üêô

Another Way to Create Intuitive Syntax for tTactics

We can avoid using syntax altogether in this particular case, and just declare the macro with the arguments fed in the appropriate places.

macro a:tactic "and_then'" b:tactic : tactic => `(tactic| ($a:tactic; all_goals $b:tactic) ) example: 1 = 1 ‚àß 2 = 2 := ‚ä¢ 1 = 1 ‚àß 2 = 2 ‚ä¢ 1 = 1‚ä¢ 2 = 2 and_then' All goals completed! üêô

This works just the same!
